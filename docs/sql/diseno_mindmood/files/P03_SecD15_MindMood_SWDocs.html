Sección 1

1 Propósito

Este documento describe la Arquitectura de Software del sistema MindMood, un Diario Personal Inteligente que analiza el estado emocional del usuario a partir de texto libre. Define la arquitectura funcional y física del sistema, las interfaces entre componentes, el comportamiento dinámico y el plan de integración, siguiendo el estándar académico CUCEI.

Sección 3

3 Restricciones y Objetivos de Realización

  • Procesar y clasificar el estado emocional en < 1.5 s (REQ-09)
  • Autenticación segura con hash bcrypt de contraseñas (REQ-01, REQ-02)
  • Almacenamiento local SQLite con cifrado AES-256 (REQ-04)
  • Modo offline-first con sincronización atómica al detectar red (REQ-11)
  • Visualizaciones interactivas: dona, línea temporal, mapa de calor
  • Exportación a CSV y PDF (REQ-13)
  • Soporte modo claro / oscuro (REQ-14)

5.6 Mecanismos de Sincronización

SQLite usa PRAGMA journal_mode=WAL para concurrencia. Operaciones de escritura usan with get_connection() as conn (Context Manager) que garantiza commit automático o rollback.

5 Descripción de Funciones y Comportamiento Dinámico

fnValidateEntry — utils.py
fnValidateEntry(strText: str) → tuple[bool, str]
DescripciónValida el texto de entrada: no vacío, ≥ MIN_WORDS (3) palabras, ≤ MAX_CHARS (10,000) chars. Función pura sin efectos secundarios.
ParámetrosstrText (input): Texto libre del usuario.
RetornoTupla (bValid: bool, strError: str). Si válido: (True, ''). Si inválido: (False, mensaje_error).
PrecondiciónstrText debe ser de tipo str.
PostcondiciónNo modifica estado. Retorno puro.
ErroresAttributeError si strText es None.
RequerimientosREQ-03
Comportamientostrip() → ¿vacío? → ¿len(split()) < MIN_WORDS? → ¿len() > MAX_CHARS?
fnAnalyzeEntry — sentiment_analysis.py
fnAnalyzeEntry(strText: str) → dict
DescripciónMotor principal NLP. Tokeniza con NLTK, aplica pysentimiento, calcula compound_mean, mood_score y detecta emociones dominantes.
ParámetrosstrText (input): Texto validado (mín. 3 palabras).
RetornodictAnalysis: compound_mean, compound_std, pos_ratio, neg_ratio, neu_ratio, mood_score, mood_label, emotions, sentences, sentence_count, dominant_emotion.
Precondiciónpysentimiento instalado y cargado. strText no vacío.
PostcondiciónRetorna análisis completo. No modifica BD.
ErroresImportError si pysentimiento no instalado. Texto muy corto reduce precisión.
RequerimientosREQ-05, REQ-06, REQ-09
ComportamientoTokenización NLTK → Loop oraciones → pysentimiento.predict() → Agregación estadística → mood_score → mood_label
fnSaveEntry — database.py
fnSaveEntry(strText: str, dictAnalysis: dict) → int
DescripciónPersiste nueva entrada en SQLite. Serializa campos JSON antes de insertar.
ParámetrosstrText (input): Texto del usuario. dictAnalysis (input): Dict de fnAnalyzeEntry.
RetornonId (int): ID de la fila insertada (AUTOINCREMENT).
PrecondiciónBD inicializada con fnInitDb(). dictAnalysis contiene mood_label y compound_mean.
PostcondiciónFila insertada en tabla 'entries'. created_at y entry_date generados con datetime.now().
Erroressqlite3.OperationalError si BD bloqueada. KeyError si dictAnalysis incompleto.
RequerimientosREQ-03, REQ-11
ComportamientoGenerar timestamps → Serializar JSON → INSERT SQL → commit → return lastrowid

fnGetAllEntries(nLimit: int = 200) → list[dict]
DescripciónConsulta todas las entradas, ordenadas DESC por created_at, limitadas por nLimit.
ParámetrosnLimit (input): Máx. filas. Default 200.
RetornolstEntries (list[dict]): Dicts con todos los campos incluyendo JSON deserializado.

fnComputeAllStats(lstEntries: list[dict]) → dict
DescripciónCalcula distribución de mood, evolución temporal, tendencia lineal e índice de estabilidad emocional.

1database.py
fnSaveEntry		 No valida unicidad de entry_date por usuario (REQ-15). Permite múltiples entradas por día. Alta Agregar UNIQUE constraint en entry_date o validación previa a INSERT.
2database.py
General		 Variables sin prefijos húngaros: 'now' → dtNow, 'conn' → dbConn, 'cursor' → dbCursor. Media Renombrar todas las variables locales aplicando notación húngara.
3app.py
fnInitAuth		 Contraseña de admin hardcodeada como 'admin123' en código fuente. Riesgo de seguridad crítico (REQ-02). Alta Mover credenciales a variables de entorno o archivo .streamlit/secrets.toml.
4app.py
fnExportToPDF		 El encode ASCII ignora acentos (text.encode('ascii','ignore')). Texto en español pierde calidad en el PDF exportado. Media Usar fpdf2 con fuente DejaVu UTF-8, o migrar a ReportLab.
5sentiment_analysis.py
fnAnalyzeEntry		 No existe manejo de timeout. Si el modelo NLP excede 1.5 s (REQ-09) no hay mecanismo de cancelación. Media Implementar threading.Timer o asyncio con timeout configurable.
6database.py
fnGenerateSyntheticData		 La función interna _generate_analysis contiene import math dentro del cuerpo. Ineficiente; se reimporta en cada llamada. Baja Mover import math al inicio del archivo (nivel de módulo).
7utils.py
fnValidateEntry		 MAX_CHARS=10,000 es inconsistente con REQ-03 (2,000 chars máx). Discrepancia entre requerimiento y código. Media Alinear MAX_CHARS=2000 con REQ-03, o actualizar el requerimiento si el cambio es intencional.
8app.py
fnRenderReminder		 datetime.strptime con formato fijo '%Y-%m-%d %H:%M:%S' no maneja timestamps con microsegundos. Baja Envolver en try/except o usar pd.to_datetime() con inferencia automática.
9database.py
General		 Falta cifrado AES-256 del campo text en SQLite (REQ-04). Actualmente el texto del diario se guarda en claro. Alta Implementar cifrado con pycryptodome o migrar a sqlcipher.
10app.py
fnRenderShareWidget		 URLs de compartir con texto sin URL-encode. Caracteres especiales (acentos, emojis) rompen los enlaces generados. Baja Aplicar urllib.parse.quote(strShareText) antes de construir la URL.

Fortalezas del Código

✅ Arquitectura clara en capas

Separación limpia entre app.py, database.py, sentiment_analysis.py, statistics.py y utils.py.

✅ Manejo de dependencias opcionales

Try/except en imports de Plotly, FPDF y streamlit-authenticator permite degradación elegante.

✅ WAL en SQLite

PRAGMA journal_mode=WAL mejora la concurrencia de lectura/escritura.

✅ Documentación interna (docstrings)

Todas las funciones principales tienen docstrings con descripción de parámetros y retorno.

✅ Datos sintéticos para testing

fnGenerateSyntheticData facilita pruebas de visualización sin datos reales.

✅ Context Manager en BD

'with get_connection() as conn' garantiza commit/rollback automático.
